## Git 与变更管理
- **禁止直接 push 到 main/master**，所有修改必须在分支上进行（`feature/xxx`、`fix/xxx`、`refactor/xxx`、`update/xxx` 等）
  > 原因：main 分支直接对应生产环境，直接 push 会跳过 code review 和 CI 验证，一旦出错无法回滚到安全状态。
- **禁止未说明影响范围就批量修改并提交**，提 PR 前必须说明本次改动的影响范围和涉及文件清单（只需在合并前说明一次，不需要每次 commit 重复）
- **一个任务 ≠ 一个 commit**（可能多个），一个 commit = 一个最小可理解变化单元，每个 commit 单一意图，message 必须表达「原因 + 改动」。commit 的边界是：一个完整、可验证、可回滚的行为变化。如果当前任务范围仍未完成，禁止为了满足流程而提前 commit。
- **变更必须局部化**，禁止"顺手优化"：改逻辑时不顺手重构整个文件，修 bug 时不顺手改无关代码，加功能时不顺手清理代码风格（除非用户明确要求）
- **规则优先级**（冲突时）：安全性 > 可追溯性 > 最小变更 > 性能 > 便利性
- **分支禁止直接 commit 到 main**，仅在用户明确说可以提交合并时才执行合并

## 测试

### 测试分层

| 层级 | 触发时机 | 执行位置 / 触发者 | 范围 |
|------|---------|-------------------|------|
| **Unit test（单元测试）** | 开发中每一步（STEP 2–4 循环内） | **本地执行** | **仅当前变更的文件/模块（增量）**，例如 `pytest tests/test_当前修改文件.py` |
| **Integration test（集成测试）** | 功能完成后（STEP 5 重构完 → STEP 6 提交前） | **本地执行**（仅当涉及多模块交互时） | 当前新增功能的端到端协作 |
>
> **「多模块交互」判定标准：**
> - ✅ 属于：修改了跨模块的接口/签名（如改了 `calculation.py` 对外暴露的函数签名，影响 `optimizer.py` 调用方）
> - ✅ 属于：修改了模块间的数据协议/数据流（如改了共享数据结构、文件格式、API 契约）
> - ❌ 不属于：纯模块内部改动（如 UI 内部展示逻辑、计算模块内部算法——前提是函数签名不变且不影响调用方行为）
> - ⚠️ 灰色地带：内部算法变化虽签名不变，但可能改变调用方行为 → 应补 Integration Test |
| **Regression test（回归测试）** | **PR 发起后、合并到 main 之前** | **CI（远程，如 GitHub Actions）自动执行** | **全量已有功能**（所有测试用例） |

### 测试纪律
- 开发中只跑相关 unit test，禁止每次跑全量
- 功能完成后补 integration test，验证端到端行为
- 改了行为就更新/补测试；不许把失败测试注释掉
- 用户说 **「完成并提PR」** 时：
  - ① 先检查是否存在未 commit 的修改；若有，列出清单并提醒用户，等待确认后再继续
  - ② push 代码并自动发起 PR，触发 CI 全量回归
  - 普通 push（未说「完成并提PR」）只做代码备份/同步，不触发 PR
- 必须通过 PR 触发 CI 全量回归。CI 变绿前，禁止合并 main
- CI 全量回归报错（变红）时：
  - ① 先分析 CI 日志定位失败用例
  - ② **在本地只跑那个失败的测试**（`pytest tests/test_xxx.py::test_func -v`）来复现和调试——这是推荐做法，避免「改一行 → 等 CI → 再改 → 再等」的恶性循环
  - ③ 修复后 push，PR 自动更新，CI 重跑
  - ❌ 禁止：在本地跑全量回归测试（`pytest` 无筛选）来复现 CI 错误。例外：当用户明确要求、CI 日志无法定位问题、或同一问题连续多次复现失败时，才允许执行全量回归
  - ❌ 禁止：不分析 CI 日志就直接改代码碰运气
